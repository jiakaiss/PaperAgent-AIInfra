## Context

当前 paper-agent 的 Web UI 是一个只读的论文浏览界面，用户通过 `localStorage` 存储偏好设置。系统采用静态配置文件（`config.yaml`）管理用户，新增用户需要手动编辑配置文件并重启服务。

现有限制：
- 用户配置是静态的，运行时无法动态添加
- Web UI 没有表单提交流程，只有前端状态管理
- Email notifier 需要预配置的 SMTP 凭据

## Goals / Non-Goals

**Goals:**
- 提供自助订阅页面，让非技术用户也能注册接收论文推送
- 持久化存储订阅信息，服务重启后不丢失
- 提升 Web UI 整体视觉质量

**Non-Goals:**
- 不实现用户认证/登录系统（保持匿名订阅）
- 不支持订阅后的偏好修改或删除（MVP 只做注册）
- 不引入新的通知渠道（只使用现有的 email notifier）
- 不实现实时推送（仍使用现有的定时任务调度）

## Decisions

### 1. 订阅数据存储方案

**决定**：扩展 SQLite 数据库，新增 `subscriptions` 表。

**理由**：
- 数据库已有 WAL 模式支持并发读写
- 比动态修改 YAML 配置文件更安全（避免格式错误、并发写入问题）
- 便于未来扩展（添加订阅时间、状态、来源等字段）

**替代方案**：
- 动态写入 config.yaml：风险高，格式错误会导致服务崩溃
- 独立的 subscriptions.json：增加复杂度，没有事务保障

### 2. 订阅到配置的桥接机制

**决定**：启动时从数据库加载订阅，合并到内存中的 AppConfig；运行时新订阅同时写入数据库和内存配置。

**理由**：
- 保持 Pipeline 等现有组件的接口不变（仍接收 AppConfig）
- 新订阅立即生效，无需重启
- 数据库作为持久化源，内存配置作为运行时缓存

**替代方案**：
- 每次 pipeline 运行前重新加载：延迟太高，且增加数据库压力
- 完全放弃 AppConfig，改用数据库：改动范围太大

### 3. Email 发送配置

**决定**：订阅用户使用全局 SMTP 配置（来自 config.yaml），不支持用户自定义 SMTP。

**理由**：
- 订阅用户是接收方，不是发送方
- 全局 SMTP 已在 config.yaml 中配置，复用即可
- 简化表单设计（只需邮箱地址，不需要 SMTP 信息）

### 4. UI 设计风格

**决定**：采用轻量级现代风格，使用 CSS 变量管理主题色，保持 HTMX 技术栈。

**理由**：
- 项目已有 HTMX + Jinja2 的技术选型，保持一致性
- CSS 变量便于未来支持深色模式
- 不引入 React/Vue 等框架，避免构建复杂度

**替代方案**：
- 引入 Tailwind CSS：增加构建步骤，对于小型项目过重
- 使用 CSS 框架（Bootstrap/Bulma）：样式定制受限，增加依赖

### 5. 表单验证策略

**决定**：客户端 + 服务端双重验证。客户端用 HTML5 原生验证 + 少量 JS；服务端用 Pydantic 模型验证。

**理由**：
- 客户端验证提供即时反馈，改善用户体验
- 服务端验证是安全底线，防止绕过客户端
- Pydantic 已有成熟的验证基础设施

## Risks / Trade-offs

- **[风险] 邮箱地址无效或不可达** → 发送验证邮件确认（可选，MVP 可跳过）；记录发送失败日志
- **[风险] 重复订阅同一邮箱** → 数据库 UNIQUE 约束 + 前端检测；返回友好提示而非错误
- **[风险] 大量订阅导致 SMTP 被标记为垃圾邮件** → 限制单次发送数量；使用专业的邮件发送服务（SendGrid 等）作为 SMTP 后端
- **[Trade-off] 不支持订阅管理（修改/删除）** → MVP 简化实现，但数据库 schema 预留 `status` 字段，未来可支持退订
- **[Trade-off] UI 改进范围有限** → 优先改进订阅页面和主页导航，其他页面渐进式优化
